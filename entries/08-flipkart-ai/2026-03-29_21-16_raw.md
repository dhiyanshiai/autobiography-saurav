Today, I shipped something small… but meaningful.

A simple app where I can speak my thoughts from my iPhone
and slowly turn them into a book.

⸻

The idea looked straightforward:

📱 Capture thoughts (voice/text)
🤖 Refine using AI
📘 Store everything in GitHub

Build consistently → compile later.

⸻

The UI came together nicely.

Minimal. Focused.
One screen. One action: capture a thought.

But then reality kicked in 🙂

⸻

First challenge:
The app didn’t work on iPhone.

Buttons wouldn’t respond.
No errors. Just… nothing.

Took me a while to realize:

Even basic JavaScript event handling behaves differently on mobile Safari.

Had to rethink how I wired interactions entirely.

⸻

Second challenge:
Saving to GitHub kept failing.

Everything worked on desktop.
But on iPhone:

👉 Status: 0
👉 Request never reached GitHub

At first, it felt like a bug.

It wasn’t.

⸻

This is where things got interesting.

iOS Safari blocks certain cross-origin requests —
especially PUT requests with authentication.

So even though the API is correct:

❌ Frontend → GitHub (write)
✅ Not reliable on mobile

⸻

That moment changed how I looked at the problem.

Instead of debugging harder,
I had to step back and redesign.

⸻

Enter: Cloudflare Workers.

New flow:

📱 App → ⚙️ Worker → 🧠 GitHub
	•	No CORS issues
	•	Secure token handling
	•	Works seamlessly from iPhone

⸻

What I thought was a small side project
ended up becoming a lesson in system design.

Not every failure is a bug.
Sometimes it’s a boundary of the platform.

⸻

Now the app works end-to-end:
	•	I can speak or type thoughts anytime
	•	Tag them, structure them
	•	Store them as entries
	•	Slowly shaping into something bigger — a book

⸻

But more importantly:

I didn’t just build a feature.
I built a way to think, capture, and reflect consistently.

⸻

Still early days.

But this feels like something I’ll keep coming back to.

⸻

Have you ever chased a “bug”…
and discovered it was actually a design limitation?
